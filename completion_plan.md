# Open2worK 補完程式計畫

更新日期：2026-03-17

## 1. 計畫目標

這份計畫針對目前 UI 測試結果，補完尚未完成的頁面行為、按鍵動作與流程一致性，目標如下：

1. 所有主要頁面的按鍵都要有明確結果：可執行、可導頁、可提示，或明確 disabled。
2. 修正路由與頁面對應錯誤，避免使用者進入錯頁。
3. 將目前「只有 UI、沒有行為」的操作補齊到最小可用版本。
4. 建立可重跑的回歸測試方式，避免之後再出現假按鍵或斷鏈。

## 2. 目前確認的缺口

### P0：立即修正

1. `/settings` 路由誤指向工作台頁面，導致設定頁無法進入。
2. 工作台頁的主要按鍵無實作：匯出報表、重新整理、查看全部。
3. 編排器允許新增 `capture_ui_tree` 步驟，但流程測試與佈署時又不支援。

### P1：高優先補完

1. 執行樹頁的分類篩選、狀態篩選、複製、刪除尚未實作。
2. 資源庫頁的篩選、模板使用、更多操作、匯出庫、批次編輯尚未實作。
3. 編排器縮放按鍵沒有行為。

### P2：體驗與維護性

1. 設定頁目前只有 toast，尚未持久化。
2. 部分頁面是展示資料，尚未串接真實資料來源。
3. 缺少按鍵層級的回歸測試與頁面 smoke test。

## 3. 實作階段

## 階段 A：修正基礎錯誤與斷鏈

目標：先讓導頁與核心流程一致。

工作項目：

1. 拆分 `/workspace` 與 `/settings` 路由，各自回傳正確模板。
2. 確認側邊欄 active 狀態可正確標示工作台與設定頁。
3. 決定 `capture_ui_tree` 的產品方向：
   - 選項 A：讓 `buildDirectPayload()` 正式支援這個步驟。
   - 選項 B：在 UI 層先標記為不可佈署，僅保留「立即試跑」。
4. 為所有無功能但可點擊的主要按鍵補上暫時行為：至少 toast、disabled 或導向。

完成標準：

1. `/settings` 可正確進入設定頁。
2. 編排器不再出現「可加入但不可跑」的不一致行為。
3. 主要頁面不再有看起來可操作、實際完全沒反應的核心按鍵。

## 階段 B：補齊最小可用功能

目標：把目前只有外觀的操作補到可用版本。

工作項目：

1. 執行樹頁
   - 分類篩選切換 UI 狀態。
   - 狀態篩選切換 UI 狀態。
   - 複製流程：先複製預設卡片資料或導到新 editor。
   - 刪除流程：先做前端確認與假資料移除。

2. 資源庫頁
   - 篩選按鍵切換篩選條件。
   - `信心度 > 80%` 切換顯示。
   - `使用` 按鍵：將模板帶入 editor 或顯示可用路徑。
   - `more_vert`：打開簡單操作選單。
   - `匯出庫` / `批次編輯`：先做最小可用版，至少可顯示待辦操作或匯出 JSON。

3. 工作台頁
   - `重新整理`：重新抓取頁面資料或至少重新渲染。
   - `查看全部`：導向完整執行列表頁，若暫無頁面則顯示說明。
   - `匯出報表`：先輸出 mock JSON / CSV。

4. 編排器頁
   - 縮小 / 放大：先改為調整節點容器 scale。
   - 清楚區分「可預覽」、「可佈署」、「僅展示」步驟。

完成標準：

1. 每頁主要按鍵都有可觀察結果。
2. 不再依賴純展示 toast 充當核心功能。
3. 使用者可完成基本的新增、測試、導向、篩選操作。

## 階段 C：設定與資料持久化

目標：把展示狀態改成真正可保存與可讀取。

工作項目：

1. 新增設定檔，例如 `assets/config/ui_settings.json`。
2. 實作設定讀寫 API，例如：
   - `GET /api/settings`
   - `POST /api/settings`
3. 設定頁載入現有值並可儲存。
4. 將執行樹頁、資源庫頁、工作台頁的假資料改為可從檔案或 API 取得。

完成標準：

1. 重新整理頁面後設定值仍存在。
2. 頁面資料不再完全寫死在模板裡。

## 階段 D：測試與驗收自動化

目標：建立可持續驗證的補完品質。

工作項目：

1. 建立頁面 smoke test：
   - 驗證 `/`, `/editor`, `/library`, `/workspace`, `/settings` 回 200。
   - 驗證關鍵文字與關鍵按鍵存在。

2. 建立按鍵覆蓋檢查：
   - 掃描模板中的 `<button>`。
   - 找出沒有 `onclick`、不是 disabled、也不是表單 submit 的按鍵。
   - 將結果作為開發檢查項。

3. 建立 API 測試：
   - `/api/run`
   - `/api/plan`
   - `/api/run-direct`
   - `/api/plan-direct`
   - `/api/template-images`
   - `/api/template-image`

4. 對高風險互動流程建立人工驗收清單：
   - ROI 框選
   - 擷取模板圖片
   - 真實滑鼠 / 鍵盤操作
   - 桌面應用切換

完成標準：

1. 每次改版可快速重跑 smoke test。
2. 新增按鍵若沒有實作，能在測試階段被提早發現。

## 4. 建議優先順序

建議依下列順序執行：

1. 階段 A：先修錯頁與流程不一致。
2. 階段 B：補主要頁面的核心按鍵。
3. 階段 C：再做設定與資料持久化。
4. 階段 D：最後補自動化測試，並回頭覆蓋 A-C。

## 6. 驗收條件

補完完成時，至少要滿足以下條件：

1. 所有主頁可正確進入正確模板。
2. 所有主要按鍵都具備下列其中一種狀態：
   - 有實作功能
   - 明確 disabled
   - 清楚標示即將推出，且不偽裝成已可用功能
3. 編排器中所有可新增步驟，行為與測試 / 佈署能力一致。
4. 設定可保存並重新載入。
5. 至少有一套 smoke test 能自動檢查頁面與關鍵 API。

## 7. 建議拆票方式

可拆成以下工作單：

1. 修正 workspace/settings 路由與側邊欄狀態
2. 補齊編排器 `capture_ui_tree` 步驟策略
3. 補完工作台主要按鍵
4. 補完執行樹篩選 / 複製 / 刪除
5. 補完資源庫主要按鍵
6. 實作設定讀寫 API 與持久化
7. 新增頁面 / API smoke tests

## 8. 風險提醒

1. 某些流程會真的操作 Windows 桌面，不能只靠 HTTP 測試判定完成。
2. ROI、滑鼠點擊、鍵盤輸入屬於高風險互動，仍需人工驗收。
3. 若先把大量假按鍵都補成 toast，而不定義真正產品行為，後續容易累積更多假功能。

## 9. 建議下一步

最合理的執行順序是：

1. 先修路由與錯頁
2. 再補核心按鍵最小可用功能
3. 最後補設定持久化與測試自動化
